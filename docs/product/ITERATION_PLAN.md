日期：2026-05-19  
关联文档：[MVP 产品方案](MVP_PRODUCT_PLAN.md)、[ADR-0011](../decisions/ADR-0011-blind-replay-primary-strategy-path.md)

## 1. 文档目的

在 [MVP 产品方案](MVP_PRODUCT_PLAN.md) 的 V0.1–V0.6 版本节奏基础上，结合**当前仓库实现状态**，给出可排期、可验收的迭代计划。  
本计划面向产品、研发与测试统一对齐「下一版做什么、不做什么、怎么算完成」。

### 1.1 交付顺序（与 MVP 文档的关系）

**近期优先：在管理后台（`apps/admin-web`）完成「盲测训练 → 一致性评估 → 行为策略 → 回测」主路径**（[ADR-0011](../decisions/ADR-0011-blind-replay-primary-strategy-path.md)），后端提供**统一、与终端无关的 REST API**（`/api/v1/...`）。已落地的开卷 K 线标注保留为**辅助能力**，不再作为主路径排期。

| 阶段 | 客户端 | 目的 |
| --- | --- | --- |
| **阶段 A** | admin-web | 快速联调、内部测试、流程与数据模型打牢 |
| **阶段 B** | 微信小程序 | 在 API 稳定、集成测试覆盖后对接同款接口，专注移动交互与微信登录 |

产品终局仍以微信小程序为面向用户的第一形态（[ADR-0004](../decisions/ADR-0004-wechat-miniprogram-first.md)）；**开发顺序**改为「后台先行」不等于改变产品定位，而是降低并行风险。

**约束**：标注 / 策略 / 回测 / 盲测的 API 从设计起即按「多端消费」建模（鉴权、用户主体、分页、错误码），避免 admin 专用接口与后续小程序接口分裂。

### 1.2 阶段 A 界面形态：管理后台内嵌移动 H5 + iPhone 17 Pro 预览壳

盲测训练、一致性报告、行为策略、回测报告等**用户向能力**在 `apps/admin-web` 中以 **移动端优先 H5** 实现（语义化 HTML、触控目标 ≥ 44px、`viewport` 适配安全区），**功能与信息架构**与微信小程序保持一致，对照 [10 微信小程序端体验](modules/10-wechat-miniprogram-experience.md) 与各模块产品文档。

| 项 | 说明 |
| --- | --- |
| 入口 | 侧栏 **「产品体验（移动 H5）」**；路由前缀 `/m/*`，嵌套于现有 `AppLayout`（左侧仍为管理菜单，右侧为手机预览区） |
| 设备壳 | **iPhone 17 Pro** 外框；**显示区域**与官方规格一致：物理 **1206×2622** 像素（竖屏 @460ppi）→ 逻辑 **402×874 pt**（@3x），实现见 `MobileIPhoneShellLayout.vue`；[Apple 技术规格](https://www.apple.com/iphone-17-pro/specs/) |
| 与小程序关系 | 阶段 A 不交付小程序工程；H5 与小程序**同款页面层级与文案**，降低阶段 B 迁移成本；API 完全一致 |
| 技术栈 | 与 admin-web 相同（Vue 3 + Element Plus）；业务页以移动布局为主，避免桌面表格阻塞触控 |

实现骨架：`layouts/MobileIPhoneShellLayout.vue`、视图 `views/mobile/*`（迭代 1–3 逐项替换占位页）。

## 4. 迭代 0：工程收口与数据就绪

**目标**：管理端与行情链路稳定，为后续标注 / 回测提供可靠数据，避免在产品开发中被基础设施打断。

### 4.1 交付物

- [ ] 合并管理端鉴权、数据查看、行情相关未提交改动；CI 通过（ruff / pytest 关键路径）
- [ ] `docker compose up` 后 API 健康检查稳定（含 `identity-service` 依赖）
- [ ] 集成测试：`登录 → 查询 K 线 → 创建导入任务`（可选）
- [ ] 核心数据覆盖检查清单：**测试环境**以创业板（`cyb`）近 6 个月日 K 为最小集（见下方命令）；全市场 ≥2 年留待生产前
- [ ] 行情性能 P0（可选）：批量在线 TS 提交合并、symbol 同步 TTL
- [ ] 生产部署检查表：Turnstile、JWT、CORS、关闭公开 OpenAPI

### 4.2 验收标准

- 管理后台可登录并完成一次「在线批量更新 + 数据查看」
- `GET` 用户侧将复用的 K 线查询路径在测试环境返回预期条数
- 无「登录无提示」「API 模块缺失导致容器 unhealthy」类阻塞问题

### 4.3 测试数据集（创业板近 6 个月）

迭代 0 / 迭代 1 联调不必先灌全 A 股，默认测试集：

| 项 | 值 |
| --- | --- |
| 市场 | `cyb`（创业板，`sz300` / `sz301`） |
| 区间 | 近 **6 个月** 至今天 |
| 周期 | **日 K**（`day`） |

**前置**：`docker compose` 已启动且 TDX 主站可用（管理后台测速至少 1 个 `ok`）。

```powershell
.\make.ps1 seed-cyb
# 可选：.\make.ps1 seed-cyb -- 在容器内加参数见下方
docker compose -f docker-compose.dev.yml exec api .venv/bin/python -m iquant_api.cli.seed_test_cyb --months 6 --periods day
```

在管理后台 **数据更新** 页查看任务进度；完成后在 **数据查看** 用 `scope=with_bars` 筛选创业板标的验收入库条数。

### 4.4 不在本迭代

- 业务级盲测 / 策略 / 回测（迭代 1 起）
- 微信小程序工程初始化可并行调研，**不列为迭代 0 交付**

## 5. 遗留交付：开卷 K 线标注（辅助，非主路径）

> **状态：已部分落地**（`annotation-service` + H5 批次标注）。按 [ADR-0011](../decisions/ADR-0011-blind-replay-primary-strategy-path.md) **不再扩展为主路径**；仅维护与可选对照。

**原目标**：开卷表达理想买卖点，持久化 `label_*` 样本。

对应模块：[01 历史 K 线标注](modules/01-historical-labeling.md)（文档已标为辅助）。

### 5.1 产品交付（admin-web 移动 H5）

| 能力 | 状态 | 说明 |
| --- | --- | --- |
| 批次标注（主路径） | ✅ | 随机 N 只有 K 线标的队列；跳过 / 完成本只；默认 20 只/轮 |
| 轮次模式总结 | ✅ | 规则描述本批标注节奏 + 用户修正项（非策略结论） |
| 历史批次与回看 | ✅ | 批次列表、明细、只读 K 线回看 |
| K 线展示 | ✅ | 触控缩放 / 平移；买/卖/撤销工具条 |
| 买卖点标注与配对 | ✅ | 先买后卖；PUT 整表替换 pairs |
| 标注收益 | ✅ | 收盘价口径 `return_pct` |
| 单只标注（次路径） | ✅ | 自选标的创建会话，可多对 |
| 意图标签 | ❌ P1 | 丰富 DSL 翻译输入，见 [01 模块 §4.7](modules/01-historical-labeling.md) |
| 合规文案 | ✅ | 非投资建议类提示 |

### 5.2 工程交付

| 层级 | 内容 |
| --- | --- |
| 新服务 | `annotation-service`（或 `labeling-service`）：标注 ORM、仓储、用例 |
| API | `/api/v1/labels/*`（或统一前缀下的资源路径；样本 CRUD、标注点 CRUD、收益试算） |
| 行情复用 | 只读调用 `market-service.query_bars`，不穿透 ORM |
| admin-web | 路由前缀 **`/m/*`**；布局 **`MobileIPhoneShellLayout`**（iPhone 17 Pro 壳）+ 业务视图；侧栏菜单「产品体验（移动 H5）」 |
| 鉴权 | **迭代 1**：可与现有**管理员 JWT**共用（单用户 / 单超管场景），但 API 设计需预留 `user_id` 或主体字段，便于阶段 B 切换为微信用户 token；**禁止**把「仅 admin 路径」写死为业务唯一入口 |
| 迁移 | `label_session`、`label_pair`、`label_batch`、`label_queue_item`、`label_batch_summary` |
| 小程序 | **本迭代不交付**；仅需保证 API 契约可被 HTTP 客户端复用 |

### 5.3 验收标准（必须全部满足）

- [ ] **admin-web** 完成不少于 **10 对**买卖标注并保存（同一会话或多会话均可）
- [ ] 任意 HTTP 客户端使用同一鉴权与路径可拉取样本列表（便于后续小程序）
- [ ] 标注与盲测无任何共用表或「混合模式」入口（遵守 ADR-0002）
- [ ] 写操作支持 **X-Idempotency-Key**（或项目既有幂等约定）
- [ ] 关键接口有集成测试或契约快照（建议从本迭代开始沉淀）

### 5.4 依赖与风险

- **依赖**：迭代 0 行情数据覆盖；日 K 优先，分钟级可列为 P1
- **风险**：移动端 K 线组件（触控命中、与柱状图对齐）需 spike；壳内视口高度需兼顾笔记本屏幕

## 6. 迭代 1：V0.1a 盲测回放（主路径起步）

**目标**：在隐藏未来 K 线下记录用户买/卖/观望，并落库每 bar 可见特征快照。

对应模块：[04 近端历史盲测回放](modules/04-blind-replay.md)

### 6.1 产品交付

- 选择标的、周期、近端区间（如 6–12 个月）
- 逐根推进，服务端裁剪可见 K 线（禁止端上泄露未来）
- 买入 / 卖出 / 观望；持仓与现金按统一成交规则模拟
- 操作理由标签、信心（P1）

### 6.2 工程交付

- `blind-replay-service`：`blind_session`、`blind_action`、特征快照 JSON
- API：`POST /replays/sessions`、`POST .../actions`、`POST .../finish`
- admin-web：**`/m/*` H5** 盲测页（复用 K 线组件，无开卷标注主入口）

### 6.3 验收标准

- [ ] 用户无法看到未来 K 线（接口 + 集成测试）
- [ ] 操作与快照可复现查询
- [ ] 与 `label_*` 表/API 隔离（ADR-0002）

## 7. 迭代 2：V0.1b 一致性评估与模式摘要

**目标**：对多轮盲测评估入场/出场/节奏一致性，输出模式摘要与训练建议；未达标不生成 DSL。

对应模块：[04](modules/04-blind-replay.md)、[05](modules/05-execution-gap-diagnosis.md)（一致性子集）

### 7.1 产品交付

- 一致性分项评分 + 矛盾样本回看
- 规则引擎摘要（描述性，非买卖建议）
- 未达标：仅训练编排建议

### 7.2 工程交付

- 用例：`evaluate_consistency`、`summarize_blind_patterns`（规则引擎，可参考 `batch_profile`）
- API：`GET /users/me/consistency` 或 `GET /replays/consistency-report`

### 7.3 验收标准

- [ ] ≥3 轮盲测后可出报告
- [ ] 文案合规（ADR-0003）

### 8.3 验收标准

- [ ] 输入仅 `blind_replay` 样本可生成合法 DSL
- [ ] 与 `label_session` 无强制关联

## 9. 迭代 4：V0.2b 回测验证、优化建议与报告

**目标**：验证行为策略是否有效，并给出可回测的优化建议。

对应模块：[03 回测验证](modules/03-backtest-validation.md)、[ADR-0009](../decisions/ADR-0009-backtest-engine-boundary.md)

### 7.1 产品交付

- 对选定 DSL 跑历史回测（先支持日 K、单标的、做多）
- 报告：收益曲线、最大回撤、胜率、交易次数、样本外一切分（时间切分即可）
- 参数敏感性（简化：1–2 个关键参数 ±10%）
- 报告页强制风险提示

### 7.2 工程交付

- `packages/backtest-engine`：撮合、持仓、费用模型（MVP 简化）
- `packages/indicators`：DSL 依赖的指标集
- Celery 任务：`strategy.run_backtest`（长任务 + 进度可选）
- API：`POST /backtests`、`GET /backtests/{id}`
- admin-web：**`/m/*` H5** 回测任务列表 + 回测详情页（图表/指标卡片）

### 7.3 验收标准

- [ ] 同一策略在固定样本上回测结果可复现
- [ ] 样本外区间指标与全样本同时展示
- [ ] 回测失败（数据不足等）有明确错误码与文案

## 10. 迭代 5：V0.3 执行偏差诊断（需行为 DSL）

**目标**：新一轮盲测中，对比**已确认行为 DSL 信号**与用户实际操作。

对应模块：[05 执行偏差诊断](modules/05-execution-gap-diagnosis.md)

### 9.1 产品交付

- 行为策略 DSL 信号与用户操作逐 bar 对齐
- 指标：信号执行率、非策略入场率、过早止盈率、止损执行率、收益损耗
- 纪律评分（加权公式 MVP 固定即可）
- admin-web：盲测复盘页、执行诊断页（小程序后续对齐）

### 9.2 工程交付

- `diagnosis-service`：对齐引擎 + 报告模型
- API：`GET /blind-sessions/{id}/diagnosis`
- PDF/分享可列为 P2

### 9.3 验收标准

- [ ] 给定一份盲测记录，报告中的偏差类型可人工核对
- [ ] 诊断仅基于可观测行为，文案避免「心理断言」

## 10. 迭代 6：V0.5 AI 解释与训练建议

**目标**：帮助用户**读懂**策略、回测与执行差距，不替代 DSL 决策。

对应模块：[07 AI 策略助手](modules/07-ai-assistant.md)

### 10.1 产品交付

- 输入：结构化报告 JSON（回测摘要 + 诊断摘要）
- 输出：自然语言解释、训练建议（偏差类型 → 训练方向映射）
- 对话式调参建议：**仅建议 DSL 参数**，改参后必须重新回测
- 内容审核 + PII 脱敏（上线前必做）

### 10.2 工程交付

- `ai-service`：编排、prompt 模板、审核钩子
- API：`POST /ai/explain`、`POST /ai/chat`（带 session 限制）
- admin-web：AI 助手页（或弹窗侧边栏；小程序后续对齐）

### 10.3 验收标准

- [ ] 任意测试用例中，AI 回复不包含直接「买入/卖出某标的」指令
- [ ] 调参建议可映射为 DSL diff 或参数字典

## 11. 迭代 7：V0.6 策略库与预警准备

**目标**：策略可版本化保存，为后续微信订阅消息预警留结构。

### 11.2 验收标准

- [ ] 用户可保存多版本策略并查看历史回测
- [ ] 预警相关表与 API 存在，但可不接真实行情推送

## 13. 明确不做（MVP 阶段）

- 实时模拟交易、实盘下单对接
- AI 直接生成买卖信号
- 多角色 RBAC（管理端保持单超管即可）
- 真实行情微信推送（迭代 7 仅结构准备）
- HQScanner 等业务外系统耦合

## 14. 成功指标（迭代级）

**阶段 A（admin-web）** 可采用**内部试用**口径（研发 + 产品 + 种子用户）：

| 迭代 | 核心指标 |
| --- | --- |
| 1–2 | 内部试用：≥3 轮盲测；一致性报告可出；操作落库成功率 ≥ 99% |

| 6 | AI 解释满意度（主观问卷 ≥ 3.5/5） |
