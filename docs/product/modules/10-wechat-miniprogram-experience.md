# 10 微信小程序端体验模块

## 1. 模块定位

微信小程序端体验模块定义 iQuant 在微信小程序中的用户体验、页面组织、交互约束和端能力边界。

iQuant 的目标用户是普通交易者，因此小程序端必须降低理解成本和操作门槛，把复杂策略、回测和执行诊断包装成清晰、可操作的流程。

## 2. 用户价值

- 用户无需安装独立 App，即可在微信生态中完成**盲测训练**、一致性/策略/回测报告查看。
- 适合碎片化使用，例如每天做一组盲测训练或查看一次策略复盘。
- 方便后续接入微信订阅消息，用于策略提醒、训练提醒和报告生成提醒。

## 3. 核心边界

小程序端负责：

- 用户交互和流程引导。
- 图表展示和触控操作。
- 策略、回测、诊断和训练结果展示。
- AI 助手对话入口。
- 微信登录、订阅消息授权和分享能力。

小程序端不负责：

- 策略生成算法。
- 回测引擎。
- 盲测执行偏差计算。
- AI 模型调用编排。
- 行情数据清洗和存储。

## 4. 小程序页面规划

| 页面 | 作用 |
| --- | --- |
| 首页 / 工作台 | 训练进度、一致性评分、待办盲测、策略状态 |
| 盲测训练页 | **主入口**：隐藏未来 K 线，买/卖/观望 |
| 一致性报告页 | 分项评分、矛盾样本、训练建议 |
| 候选策略页 | 行为策略规则列表，确认 DSL |
| 回测详情页 | 收益、回撤、样本外、风险提示 |
| 优化建议页 | 参数/规则/训练建议（附回测依据） |
| 盲测复盘页 | DSL 信号 vs 用户操作（二期） |
| 执行诊断页 | 纪律评分、收益损耗 |
| 训练任务页 | 专项盲测练习 |
| 历史标注页 | **辅助**，开卷对照 |
| AI 助手页 | 解释报告与建议 |
| 策略库页 | 策略版本 |
| 我的页 | 登录、订阅、设置 |

## 5. 交互设计原则

### 5.1 移动端优先

- 所有核心操作必须适配单手触控。
- K 线标注需要支持点击选中、选择意图标签、确认标注、撤销和删除。
- 不依赖鼠标悬停能力。
- 避免复杂表格，关键指标要卡片化、分组化展示。

### 5.2 流程分步

普通用户不应一次面对过多概念。

建议将复杂流程拆成：

```text
盲测训练（多轮）
  -> 查看一致性报告
  -> 确认行为策略
  -> 回测与优化建议
  ->（可选）新一轮盲测 vs DSL → 执行偏差
```

### 5.3 结果要可解释

小程序端的每个结论都应给出“为什么”：

- 为什么生成这个策略。
- 为什么认为有过拟合风险。
- 为什么判定用户提前买入。
- 为什么建议做某类训练。

### 5.4 风险提示前置

交易类产品不能暗示收益保证。

关键位置需要提示：

- 本产品是策略验证和训练工具，不构成投资建议。
- 历史回测不代表未来收益。
- 盲测训练不能完全替代真实交易心理压力。

## 6. 小程序端能力约束

### 6.1 K 线图

需要重点验证：

- 大量 K 线数据在小程序 canvas 中的性能。
- 缩放、拖拽、点击选中 K 线的流畅度。
- 买卖点标注在小屏幕上的可用性。
- 指标叠加时的信息密度。

MVP 可以先限制：

- 日线周期。
- 单标的。
- 最近 6 到 24 个月数据。
- 指标数量默认较少，只展示核心指标。

### 6.2 计算能力

小程序端只做轻量计算：

- 当前展示状态。
- 轻量收益展示。
- 简单交互校验。

重计算放在后端：

- 指标批量计算。
- 策略生成。
- 回测。
- 执行偏差诊断。
- AI 分析。

### 6.3 微信能力

可利用：

- 微信登录。
- 订阅消息。
- 分享训练结果或策略报告摘要。
- 云开发或自建后端 API。

MVP 优先使用自建后端 API，避免核心策略数据与计算能力过度绑定小程序端。

## 7. 输入与输出

输入：

- 用户触控操作。
- 用户标注。
- 用户盲测买卖操作。
- 用户 AI 对话输入。
- 微信登录与授权状态。

输出：

- 标注样本。
- 盲测操作记录。
- 策略和回测报告展示。
- 执行诊断报告展示。
- 训练任务展示。
- 订阅消息授权状态。

## 8. 验收标准

- 用户能在小程序端完成一次历史买卖点标注。
- 用户能在小程序端完成一次盲测回放。
- K 线拖拽、缩放和点击标注在主流手机上可用。
- 小程序端不直接承载策略生成、回测和 AI 推理核心逻辑。
- 所有交易相关页面都有必要风险提示。
