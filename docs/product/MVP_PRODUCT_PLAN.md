# iQuant MVP 产品方案

版本：V0.1  
日期：2026-05-13

## 1. 产品定位

iQuant 面向有交易想法但不会编程的个人交易者、投资学习者和交易培训场景。MVP 用户端形态为微信小程序，服务端提供策略生成、回测、盲测回放、执行偏差诊断和 AI 辅助能力。

产品通过“历史 K 线标注 + 理想策略生成 + 回测验证 + 近端历史盲测回放 + 执行偏差诊断 + AI 解释训练”，帮助用户完成两件事：

1. 验证自己理想中的交易策略是否真的具备盈利能力。
2. 找到自己实际交易行为与理想策略之间的差距，并通过训练缩小这段差距。

一句话定位：

> 帮助用户把交易想法变成可验证的策略，并把策略执行从“知道”推进到“做到”。

## 1.1 产品形态

MVP 采用微信小程序作为用户端。

产品形态约束：

- 普通用户通过微信小程序完成标注、盲测、报告查看和 AI 对话。
- 策略生成、回测、执行诊断和 AI 分析放在服务端。
- 小程序端只承担交互呈现、轻量状态管理和必要的本地校验。
- 核心业务能力保持平台无关，后续可扩展到 Web 或 App。
- 后续真实行情预警可优先接入微信订阅消息。

## 2. 核心客户价值

### 2.1 验证理想策略是否有效

用户通过完整历史 K 线标注买卖点，本质是在表达“如果我理想执行，我希望这样交易”。

系统将这些标注转化为结构化理想策略，并通过回测、样本外验证、参数敏感性测试来回答：

- 这套理想策略在历史上是否赚钱？
- 收益是否稳定，还是依赖少数几笔大交易？
- 最大回撤是否可承受？
- 换一个时间段或市场阶段是否仍然有效？
- 参数稍微变化后是否立刻失效？

### 2.2 诊断实际执行与理想策略的差距

历史盲测回放不用于生成理想策略，而用于验证用户在看不到未来走势时是否能执行自己的理想策略。

系统将用户盲测操作与理想策略信号逐条对齐，识别：

- 提前买入
- 错过买点
- 追高入场
- 过早止盈
- 止损不执行
- 频繁交易
- 持仓周期偏离
- 执行导致的收益损耗

### 2.3 训练用户缩小执行差距

产品不止输出报告，还要提供针对性训练。

例如：

- 用户经常提前买入：安排更多假突破行情训练。
- 用户经常过早止盈：安排趋势延续行情训练。
- 用户止损不执行：强化风险边界和纪律训练。
- 用户频繁交易：训练空仓等待能力。

## 3. MVP 核心原则

### 3.1 历史标注与盲测回放不能混合

两类行为的用户心态完全不同：

- 历史 K 线标注：用户已经看到后续走势，是“开卷表达”，用于表达理想策略。
- 历史盲测回放：用户看不到未来走势，是“类实战决策”，用于诊断执行偏差。

因此 MVP 必须保持两条链路独立：

```text
历史 K 线标注
  -> 生成理想策略
  -> 回测验证理想策略盈利能力

近端历史盲测回放
  -> 对比理想策略信号与用户实际操作
  -> 诊断执行差距
  -> 训练改进
```

### 3.2 AI 不直接决定交易信号

AI 的职责是解释、总结、诊断和辅助调参。

买卖信号必须由确定性的策略 DSL、指标计算和回测引擎产生。

### 3.3 策略必须结构化

系统不能直接生成散乱代码。理想策略需要转成统一 DSL，用于：

- 回测
- 盲测信号对齐
- 策略解释
- 参数调整
- 策略版本管理
- 后续预警

### 3.4 MVP 暂不做实时模拟交易

受行情接口限制，第一阶段不做真正实时模拟交易。

MVP 使用最近历史行情做盲测回放：

- 使用最近 6 到 24 个月历史数据。
- 隐藏未来 K 线。
- 用户逐根或按步长推进。
- 交易按统一成交规则模拟。

### 3.5 小程序端不承载核心计算

小程序端需要保证触控体验和使用门槛，但不能把复杂计算放到端上。

端上只做：

- K 线展示和交互状态。
- 买卖点标注交互。
- 盲测操作提交。
- 报告展示。
- AI 对话入口。

服务端负责：

- 指标计算。
- 策略生成。
- 回测。
- 执行偏差诊断。
- AI 调用编排。

## 5. MVP 用户主流程

### 5.1 理想策略验证流程

```text
选择标的与周期
  -> 加载历史 K 线
  -> 用户标注理想买点和卖点
  -> 系统提取标注特征
  -> 生成 3 到 5 个候选理想策略
  -> 用户选择并微调策略
  -> 回测验证
  -> 保存理想策略版本
```

### 5.2 执行偏差诊断流程

```text
选择一个已保存理想策略
  -> 选择近端历史盲测区间
  -> 系统隐藏未来 K 线
  -> 用户买入 / 卖出 / 观望
  -> 系统记录实际操作与当时策略信号
  -> 盲测结束后生成偏差诊断
  -> 输出纪律评分、收益损耗和训练建议
```

### 5.3 训练改进流程

```text
读取执行偏差报告
  -> 找到主要偏差类型
  -> 系统安排专项盲测训练
  -> 用户重复练习
  -> 对比训练前后执行评分和收益损耗
```

## 6. 微信小程序关键页面

| 页面 | 作用 |
| --- | --- |
| 工作台 | 展示策略、盲测任务、执行评分和待训练问题 |
| 策略创建页 | 选择标的、周期、历史区间，进入标注流程 |
| 历史标注页 | 在完整历史 K 线上标注理想买卖点 |
| 候选策略页 | 展示候选理想策略、规则解释和初步验证 |
| 回测详情页 | 展示收益、回撤、样本外表现和风险提示 |
| 盲测回放页 | 隐藏未来 K 线，用户执行买卖操作 |
| 盲测复盘页 | 按交易展示用户操作与理想策略信号的偏差 |
| 执行诊断页 | 展示纪律评分、偏差类型和收益损耗 |
| 训练任务页 | 针对偏差类型安排专项盲测训练 |
| AI 助手页 | 解释策略、回测和执行偏差，辅助调参 |
| 策略库页 | 管理策略版本、回测报告和后续预警配置 |
| 我的页 | 管理登录、授权、订阅消息和用户设置 |

小程序页面设计原则：

- 移动端优先，不依赖鼠标悬停。
- K 线标注必须适配触控点击、撤销、确认和删除。
- 复杂指标用分组卡片展示，避免大表格堆叠。
- 关键交易相关页面必须展示风险提示。

## 7. 版本节奏

### V0.1 历史标注与理想策略表达

目标：用户能表达自己的理想买卖逻辑。

交付：

- K 线展示
- 买卖点标注
- 标注配对
- 标注收益计算
- 标注样本管理
- 小程序端 K 线触控标注体验验证

### V0.2 理想策略生成与回测验证

目标：验证理想策略是否有盈利能力。

交付：

- 指标计算
- 特征提取
- 模板匹配
- 策略 DSL
- 回测引擎
- 样本外验证

### V0.3 近端历史盲测回放

目标：让用户在隐藏未来走势下完成类实战交易。

交付：

- 隐藏未来 K 线
- 逐根回放
- 买入 / 卖出 / 观望
- 成交规则
- 盲测交易记录
- 小程序端逐根回放和买卖按钮交互

### V0.4 执行偏差诊断

目标：判断用户是否能执行自己的理想策略。

交付：

- 策略信号与用户操作对齐
- 信号执行率
- 非策略入场率
- 过早止盈率
- 止损执行率
- 收益损耗
- 纪律评分

### V0.5 AI 解释与训练建议

目标：让用户看懂策略、回测和执行差距。

交付：

- 策略自然语言解释
- 回测结果解读
- 执行偏差总结
- 训练建议
- 对话式调参建议

### V0.6 策略库与预警准备

目标：为后续真实行情预警做结构准备。

交付：

- 策略保存
- 策略版本管理
- 信号记录结构
- 预警配置模型
- 微信订阅消息授权模型

## 8. 成功指标

### 8.1 产品使用指标

- 完成 10 对历史标注的用户比例
- 从标注到生成候选策略的转化率
- 用户保存理想策略的比例
- 完成一次盲测任务的比例
- 查看执行偏差报告的比例
- 完成专项训练的比例

### 8.2 策略验证指标

- 理想策略样本外表现
- 策略交易次数是否足够
- 参数敏感性是否稳定
- 最大回撤是否可接受
- 用户是否愿意保留该策略继续盲测

### 8.3 执行训练指标

- 策略信号执行率是否提升
- 非策略入场率是否下降
- 过早止盈率是否下降
- 止损执行率是否提升
- 用户盲测收益与理想策略同期收益差距是否缩小

## 9. 风险与边界

| 风险 | 应对方式 |
| --- | --- |
| 用户误以为系统保证盈利 | 明确产品是策略验证与训练工具，不承诺收益 |
| 历史标注导致过拟合 | 强制做样本外验证和参数敏感性测试 |
| 盲测不等于真实交易 | 明确盲测只能降低后视偏差，不能完全替代实盘压力 |
| AI 输出误导 | AI 只做解释和建议，策略修改必须转 DSL 并回测 |
| 执行偏差被过度心理化 | 只基于可观测行为做诊断，避免武断判断用户内心 |

## 10. 产品表达

不建议将产品包装成“AI 炒股神器”。

建议表达：

> 通过历史标注建立你的理想交易策略，用回测验证它是否具备盈利能力，再通过隐藏未来走势的盲测训练，发现并缩小你真实交易与理想策略之间的执行差距。
