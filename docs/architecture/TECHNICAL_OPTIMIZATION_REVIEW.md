# 技术规划优化审查

日期：2026-05-14

## 1. 当前目标修正

当前阶段目标从“直接面向 10 万用户设计生产形态”调整为：

> MVP 稳定支撑 1 万注册用户，并保留向 10 万用户平滑扩容的路径。

设计优先级：

1. 业务正确性：策略 DSL、回测、盲测遮蔽、执行诊断必须可复现、可审计。
2. 简单高性能：优先用成熟组件和模块化单体，避免过早微服务化。
3. 弹性扩容：API、Worker、数据库、缓存和行情存储都预留独立扩容路径。
4. 成本可控：MVP 不默认上 Kubernetes、数据库只读副本、独立 TimescaleDB 或 RabbitMQ。

## 2. 审查结论

| 领域 | 原规划问题 | 优化决策 |
| --- | --- | --- |
| 容量规划 | 直接按 10 万注册用户和 8k~12k DAU 估算，MVP 成本偏高 | 改为 MVP 1 万注册用户、1k~2k DAU，并给出阶段化扩容路径 |
| 部署 | 默认生产 Kubernetes、多 worker deployment、HPA | MVP 采用 Docker/轻量容器服务 + 托管 PostgreSQL + Redis；K8s 延后 |
| 数据库 | 默认业务库和 TimescaleDB 物理分离、只读副本 | MVP 单 PostgreSQL 实例，可启用 TimescaleDB 扩展；数据量或查询压力触发后再拆 |
| 消息队列 | 默认多队列、多 worker 池、RabbitMQ 中期迁移 | MVP 3 条队列即可：default、cpu、io；Redis broker 先满足 |
| 缓存 | Redis 4GB 主从起步，偏重 | MVP 1~2GB 起步，热点 K 线和报告摘要优先缓存 |
| 回测/诊断 | 有长任务风险，但方向正确 | 保留异步化、幂等和限额；MVP 限制任务窗口和并发 |
| AI | 可能成为成本与延迟瓶颈 | AI 默认异步/可降级；按用户配额限制；不阻塞核心策略结果 |

## 3. 保留的正确设计

- 模块化单体起步，代码边界按服务职责拆分。
- API 与 Worker 分进程，重任务异步化。
- 策略 DSL 作为核心契约。
- 历史标注与盲测回放严格分离；**主产品路径**以盲测归纳行为策略（[ADR-0011](decisions/ADR-0011-blind-replay-primary-strategy-path.md)）。
- AI 不直接决定交易信号。
- PostgreSQL 作为业务主库，Redis 用于缓存、限流、短期状态和任务 broker。
- 回测、诊断、AI 调用必须有幂等键、超时、重试和降级。

## 4. MVP 关键容量假设

| 指标 | MVP 目标 |
| --- | --- |
| 注册用户 | 10,000 |
| MAU | 3,000 ~ 5,000 |
| DAU | 1,000 ~ 2,000 |
| 峰值活跃在线 | 200 ~ 500 |
| 峰值 API QPS | 100 ~ 200 |
| 回测任务峰值 | 10 ~ 20 / 分钟 |
| 诊断任务峰值 | 20 ~ 40 / 分钟 |
| AI 请求峰值 | 10 ~ 20 / 分钟 |

## 5. MVP 推荐生产形态

```text
微信小程序
  -> 云 SLB / WAF
  -> API 容器 × 2
  -> Worker 容器 × 2~4
  -> Beat 容器 × 1
  -> 托管 PostgreSQL 单主
  -> 托管 Redis 单主或主从
  -> 对象存储
```

说明：

- API 与 Worker 分离，便于独立扩容。
- 数据库使用托管服务，避免团队早期承担备份、故障转移和磁盘运维。
- Redis 可以先单主，达到稳定付费用户规模后切主从。
- Kubernetes、只读副本、独立 TimescaleDB、RabbitMQ 都是明确触发后的扩容项。

## 6. 扩容触发条件

| 触发条件 | 扩容动作 |
| --- | --- |
| API P95 连续 3 天 > 300ms 且 CPU > 60% | 增加 API 副本 |
| Worker 队列 pending > 500 持续 10 分钟 | 增加 Worker 或拆分队列 |
| 回测 P95 完成时间 > 60s | 增加 CPU Worker，限制单任务窗口，优化回测引擎 |
| PostgreSQL CPU > 60% 或慢查询增加 | 补索引、优化查询；仍不足再加只读副本 |
| 行情表 > 200GB 或明显影响业务查询 | 拆出独立行情库 / TimescaleDB |
| Redis 内存 > 70% 或连接数接近上限 | 扩内存、主从或分离 broker/cache |
| AI 成本超过预算 80% | 限额、缓存解释、降级模板、模型分层 |
| DAU > 5,000 且多模块资源曲线差异明显 | 考虑 K8s 或按服务独立 deployment |

## 7. 后续落地要求

- 容量、部署、数据存储、异步任务文档以本审查结论为准。
- 每次从 MVP 形态升级到下一阶段，都需要压测数据和 ADR。
- 新增复杂基础设施前必须先证明当前瓶颈真实存在。
