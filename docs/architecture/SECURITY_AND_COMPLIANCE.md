# 安全与合规

本文定义 iQuant 服务端的安全设计与合规边界。安全要求按"默认拒绝、最小权限、纵深防御"组织。

## 1. 威胁模型

主要威胁：

- 凭据泄漏：openid 被滥用、JWT 被盗用。
- 越权：用户 A 访问/修改用户 B 的策略、报告、盲测会话。
- 滥用：自动化脚本恶意刷接口（注册、回测、AI 对话）。
- 投毒：恶意 K 线数据、恶意 Prompt 注入。
- 内容风险：用户输入或 AI 输出涉及违规、误导内容。
- 监管风险：未注明风险提示、被误解为投顾或荐股。
- 数据泄漏：备份外泄、运维人员越权访问、对象存储桶错配。

## 2. 鉴权

### 2.1 微信小程序登录

1. 小程序 `wx.login()` 获取 `code`。
2. 调用 `POST /auth/wechat-login`，body 含 `code` 与可选用户信息。
3. 服务端使用微信 `code2session` 接口换取 `openid`、`unionid`、`session_key`。
4. 服务端创建或更新 `user` 记录。
5. 服务端签发 access_token（15 分钟）与 refresh_token（30 天）。
6. `session_key` 仅在服务端持有，不传给小程序。

### 2.2 JWT 设计

- Access token：JWT（HS256 起步），载荷只含 `sub=user_id`、`exp`、`iat`、`jti`。
- Refresh token：随机 256bit，不放敏感信息；服务端存 hash 在 `auth_session` 表。
- 撤销：将 jti 或 refresh token hash 写入吊销集合，TTL = token 剩余有效期。
- 密钥：HS256 密钥从 KMS / 环境变量注入；每 90 天轮换。
- 高敏感操作（修改预警、删除策略）需要"短期重认证"或二次校验（小程序面部/口令均无对应能力 → 用风险提示 + 二次确认按钮，或微信生物授权）。

### 2.3 不允许

- 把 access token 写到查询参数。
- 把 token 持久化到对象存储或日志。
- 用同一个 token 横跨多个 openid。

## 3. 授权与隔离

- 所有用户数据查询必须在 service 层强制注入 `user_id` 条件，禁止 API 路由侧自行拼 SQL。
- 任何返回资源详情的接口都要先做"归属校验"，不归属于当前用户：返回 404，避免枚举。
- 管理后台与用户接口物理隔离：管理 API 走独立路径 + 独立鉴权 + 独立审计。
- 行级安全（RLS）：PostgreSQL 启用 RLS 作为兜底，业务代码失误时也无法越权（B 端版本必须开启）。

## 4. 接口安全

- HTTPS only；HTTP 严格重定向。
- HSTS：`max-age=31536000; includeSubDomains; preload`。
- CSP：管理后台启用严格 CSP；API 不返回 HTML 因此 CSP 影响小。
- CORS：API 仅允许小程序业务域名（小程序请求带特定 UA 与签名，普通跨域请求拒绝）。
- 防 CSRF：JWT 走 Authorization Header 不走 Cookie，天然规避；管理后台若用 Cookie，强制 SameSite=Lax + CSRF token。
- 防爬：限流（API_DESIGN §10）+ 设备指纹（小程序提供 deviceId）+ 异常行为风控。

## 5. 输入与校验

- 所有外部输入走 Pydantic 严格 schema：未知字段拒绝、类型严格、长度上限、正则约束。
- DSL、Prompt 等结构化输入必须经过完整 schema 验证；不允许"半结构 + 文本"。
- 上传内容只允许预签名 URL 直传对象存储，禁止经过 API 中转大文件。
- SQL 注入：禁止字符串拼 SQL；ORM + 参数绑定；`text()` 必须传参数对象。
- 命令注入：禁止用 `subprocess` 执行任何含用户输入的命令。
- 反序列化：不接受 `pickle`；JSON 反序列化使用受控解析器（限制深度、长度）。

## 6. 敏感数据保护

| 数据 | 处理 |
| --- | --- |
| openid / unionid | 入库前不哈希，但日志/链路追踪中必须脱敏（前 4 后 4） |
| 手机号 / 邮箱（如未来收集） | 入库前加密（AES-GCM），展示用掩码 |
| refresh_token | 仅存 SHA-256 hash |
| 第三方 API Key | KMS / 环境变量；禁止入仓库 |
| 数据库密码 | KMS / 环境变量；通过 Secret Manager 拉取 |
| AI 对话内容 | 默认仅在用户上下文可见，运维只在审计场景下经审批临时访问 |

### 6.1 日志脱敏

- 框架级日志拦截器统一脱敏 openid / unionid / token / 手机号 / 邮箱。
- 任何 `repr()` 用户对象时必须经过脱敏过滤器。
- 异常堆栈不得包含用户原始输入；必要时只记录哈希。

### 6.2 加密

- 传输：TLS 1.2+（建议 TLS 1.3）。
- 存储：磁盘加密由云盘保障；敏感字段在应用层 AES-GCM 加密，密钥版本化。
- 备份：备份件加密 + 桶访问最小权限。

## 7. 速率与配额

- 全局限流：API 网关层做 IP 级限流，5xx 期间自动降级。
- 用户级限流：见 API_DESIGN §10。
- 高成本资源（回测、AI）配额：免费用户每日 N 次，付费用户配额更高。
- 配额：Redis 滚动计数器；监控告警发现异常用户。

## 8. 内容审核

### 8.1 用户输入

- 标注备注、策略名称、AI 用户消息均经过内容审核：
  - 微信 `msg_sec_check` 接口（小程序内容安全 API）。
  - 自研敏感词库（金融违规 / 政治 / 色情 / 暴恐）。
- 命中：
  - 违规等级高：直接拒绝并提示。
  - 违规等级中：记录但允许（视产品策略）；触达频次升高后限制。

### 8.2 AI 输出

- 大模型输出必须 100% 经过审核：违规直接替换为安全模板。
- 关键事实声明（涉及收益预测、个股推荐）必须有显式风险提示。
- 不允许 AI 给出"买入 X 股票"的明确指令；模型 prompt 中显式禁止。
- 审核结果记录到 `moderation_result` 字段，留痕。

## 9. 风险提示与合规边界

- 所有涉及交易策略、回测、信号的页面与 API 返回均必须包含/可关联到风险提示。
- 用户首次使用需勾选"风险已知"，记录到 `user.risk_acknowledged_at`。
- 严禁文案：保证收益、稳赚不赔、专业投顾、个股推荐、操盘指导。
- 推荐文案见 [MVP_PRODUCT_PLAN §10](../product/MVP_PRODUCT_PLAN.md#10-产品表达)。
- 投顾资质：MVP 严格定位为"策略验证与训练工具"，不提供投顾服务。如未来扩展投顾业务，必须前置合规审查。

## 10. 数据合规

- 用户授权：注册环节出示《用户协议》《隐私政策》，明确数据用途。
- 数据最小化：只收集业务必要数据（openid、unionid、用户主动提交内容）。
- 用户权利：提供"删除我的数据"入口；后端 30 天内完成清理（含对象存储）。
- 数据出境：MVP 部署境内（中国大陆），不涉及出境。如未来出海，需符合 PIPL/GDPR。
- 行情数据：仅使用合规授权数据源，不自行爬取受限数据。
- 第三方 AI 数据：用户输入会被发送给第三方大模型，需在隐私政策中明示。

## 11. 三方供应商

| 供应商 | 用途 | 风险 | 应对 |
| --- | --- | --- | --- |
| 微信开放平台 | 登录、订阅消息、内容审核 | 接口变更 | 多版本兼容、监控降级 |
| 行情供应商 | K 线数据 | 数据质量、限流 | 多源、本地缓存、异常告警 |
| 大模型供应商 | AI 解释 | 内容、合规、稳定性 | 多供应商、模板兜底 |
| 云厂商（OSS / KMS / K8s） | 基础设施 | 故障、计费 | 多可用区、预算告警 |

合同与 DPA（数据处理协议）由法务统一管理。

## 12. 运维与权限

- 生产数据库禁止开发直连；只有值班 SRE + 受限角色可走堡垒机。
- 任何生产查询/变更需走审批流，审计日志留存 180 天。
- 服务账号最小权限（IAM 策略），按"读 / 写 / 管理"分级。
- 镜像不内置 SSH；调试通过 K8s exec + 审计。
- 密钥仅通过 Secret Manager 注入到 Pod 环境变量，不挂载明文文件。

## 13. 漏洞与依赖管理

- 依赖扫描：CI 跑 `pip-audit`、镜像跑 `trivy`，高危漏洞阻断合并。
- 上游依赖：依赖锁定到精确版本；不允许 `==latest` 或浮动版本。
- 漏洞响应：发现高危漏洞 24 小时内补丁；中危一周内；低危按月。
- 第三方 SDK：进入前做安全评审，离开时清理残留。

## 14. 审计

- 鉴权失败、权限拒绝、风险操作必须记录审计日志。
- 关键字段（用户 ID、IP、时间、动作、目标资源）入审计表 + 对象存储归档。
- 审计保留：180 天热存、3 年冷存。
- 内部访问（管理员越权查询）双因子 + 审计。

## 15. 应急响应

- 安全事件分级：P0（账号大规模泄漏）/P1（个别越权）/P2（异常滥用）/P3（情报）。
- P0/P1：30 分钟内启动响应流程；保全证据 → 控制扩散 → 修复 → 复盘。
- 用户通知：依据合规要求，重大事件 72 小时内通知用户与监管。
- 公开预案文档：`docs/ops/incident-runbooks/`（后续补充）。

## 16. 反例

- ❌ 在前端做归属校验（必须服务端校验）。
- ❌ 把 openid 当作只用 openid 鉴权令牌。
- ❌ 通过 query string 传 token。
- ❌ "管理员可以看所有用户内容"无审计。
- ❌ 用户主动注销后只是改 `deleted_at` 不清理对象存储。
- ❌ AI Prompt 中拼接用户输入而未做注入防御。
- ❌ 接口对未授权请求返回 200 + 空数据，掩盖未授权访问。
