# 系统架构总览

本文从模块、依赖、数据流三个维度描述 iQuant 服务端的整体架构。配合 [PROJECT_STRUCTURE.md](PROJECT_STRUCTURE.md) 阅读。

## 1. 架构原则

- **模块化单体起步**：MVP 阶段采用模块化单体，先稳定支撑 1 万注册用户；增长到 10 万用户时按真实瓶颈拆分。
- **包层无副作用**：`packages/*` 纯算法 + 纯领域模型，可被任意应用层和服务层组合使用，便于单测和未来拆分。
- **应用层薄**：`apps/api` 只负责协议适配、鉴权、参数校验、用例编排；`apps/worker` 只负责任务调度与执行。
- **稳定契约**：策略 DSL、K 线模型、盲测会话、回测报告、诊断报告是模块间唯一允许跨边界传输的数据结构。
- **服务无状态**：API 进程不持有会话状态。需要"上下文感知"的功能（盲测会话推进、AI 多轮对话）通过 Redis + PostgreSQL 显式建模为状态机。
- **重任务异步化**：API 不同步执行回测、诊断和 AI 调用。任何超过 500ms 的可预期路径必须拆为任务队列。
- **不提前复杂化**：MVP 不默认 Kubernetes、独立行情库、只读副本、RabbitMQ 或分库分表。

## 3. 模块边界

服务层职责见 [PROJECT_STRUCTURE.md §5、§6](PROJECT_STRUCTURE.md)。各服务通过以下契约协作：

| 上游 → 下游 | 契约 | 同步/异步 |
| --- | --- | --- |
| 标注 → 策略服务 | 标注样本 + 标的元数据 | 同步 |
| 策略服务 → 回测服务 | StrategyDSL + 回测参数 | 异步（任务） |
| 回测服务 → 策略服务 | BacktestReport | 异步（任务回调） |
| 盲测服务 → 诊断服务 | BlindReplaySession + 用户操作流 | 异步（盲测结束触发） |
| 诊断服务 → AI 服务 | ExecutionDiagnosisReport | 同步或异步 |
| AI 服务 → 任意服务 | 只读结构化输入，输出建议而非动作 | 同步 |
| 所有服务 → 行情/样本服务 | symbol + 周期 + 区间 | 同步（命中缓存） |

跨服务调用一律走包层接口或服务层 Python 函数调用（单体阶段），不能直接相互读对方的数据库表。

## 4. 关键链路

### 4.1 历史标注 → 理想策略 → 回测

```text
小程序
  ├─ POST /labels/sessions               创建标注会话
  ├─ POST /labels/sessions/{id}/marks    提交买卖点
  └─ POST /strategies/from-labels        触发策略生成
       └─ strategy-service
            ├─ 拉取标注样本
            ├─ 调用 packages/indicators 抽特征
            ├─ 调用 packages/strategy-dsl 输出候选 DSL
            └─ 返回 3~5 个候选策略 ID

小程序
  └─ POST /backtests                     触发回测
       └─ backtest-service
            ├─ 入库回测任务
            ├─ 投递 Celery 任务 run_backtest
            └─ 返回 task_id

worker
  └─ run_backtest 任务
       ├─ 拉取 DSL + 行情
       ├─ 调用 packages/backtest-engine
       ├─ 写入 BacktestReport
       ├─ 缓存关键指标到 Redis
       └─ 通过 Redis pub/sub 通知 API（或前端轮询）

小程序
  └─ GET /backtests/{task_id}            查询结果
```

### 4.2 盲测回放 → 执行偏差诊断

```text
小程序
  ├─ POST /replays/sessions              开启盲测会话
  ├─ POST /replays/sessions/{id}/step    推进一根 K 线
  ├─ POST /replays/sessions/{id}/actions 提交买/卖/观望
  └─ POST /replays/sessions/{id}/finish  结束盲测

API → replay-service
  ├─ 每一步只下发"当前可见 K 线截止时间"
  ├─ 未来 K 线在服务端完全隐藏（不传给前端）
  └─ 操作记录写入 PostgreSQL（带时间戳与可见快照哈希）

盲测结束 → 触发 Celery 任务 run_diagnosis
  └─ diagnosis-service
       ├─ 加载策略 DSL、用户操作、对应区间 K 线
       ├─ 调用 packages/diagnosis-engine 计算偏差
       ├─ 写入 ExecutionDiagnosisReport
       └─ 触发 ai-service 生成解释（可异步）
```

### 4.3 AI 解释

```text
小程序
  └─ POST /ai/explanations               请求解释
       └─ ai-service
            ├─ 仅消费结构化报告（DSL / 回测 / 诊断）
            ├─ 渲染 Prompt 模板
            ├─ 调用大模型（带超时与降级）
            ├─ 结构化解析返回值（Pydantic 校验）
            ├─ 持久化对话与上下文
            └─ 返回结构化解释 + 可读文案
```

AI 不能调用任何会改变交易语义的接口；服务层在路由层强制隔离。

## 5. 依赖方向（强约束）

```text
apps/miniprogram
       │
       ▼
apps/api  ──────────►  services/*  ──────────►  packages/*
   │                        │
   ▼                        ▼
apps/worker  ──────────►  services/*  ──────────►  packages/*
                            │
                            ▼
                       packages/domain
```

禁止：

- `packages/domain` 反向引用 `apps/*` 或 `services/*`。
- `packages/backtest-engine` 引用 `apps/miniprogram`。
- `packages/diagnosis-engine` 引用 `packages/ai-assistant`。
- `services/*` 互相直接读对方表（必须通过对方服务接口）。
- `apps/api` 直接调用 `packages/*` 而绕过 `services/*`（个别只读查询除外，但需登记白名单）。

CI 中使用 `import-linter` 强制依赖方向。

## 6. 数据流

### 6.1 主要数据实体

| 实体 | 主存 | 缓存 | 备注 |
| --- | --- | --- | --- |
| 用户/账户 | PostgreSQL | Redis（会话） | openid/unionid 索引 |
| K 线 | PostgreSQL 分区表；可启用 TimescaleDB 扩展 | Redis（近端窗口） | MVP 不默认拆独立行情库 |
| 指标缓存 | PostgreSQL / Redis | Redis | 计算昂贵的常用指标按版本缓存 |
| 标注会话 | PostgreSQL | - | 含标注序列与配对结果 |
| 策略 DSL | PostgreSQL | Redis（最近使用） | 不可变版本 + 草稿 |
| 回测任务/报告 | PostgreSQL（元数据）+ 对象存储（明细 JSON） | Redis（指标摘要） | 任务幂等键 |
| 盲测会话 | PostgreSQL | Redis（推进游标） | 操作记录追加写 |
| 诊断报告 | PostgreSQL + 对象存储 | Redis（摘要） | 与盲测会话 1:1 |
| AI 对话 | PostgreSQL（消息流）+ 对象存储（长上下文） | Redis（最近 N 轮） | 内容审核日志单独存 |

### 6.2 数据生命周期摘要

- 行情：长期保留，MVP 用 PostgreSQL 分区和对象存储归档；数据量触发后再拆 TimescaleDB。
- 标注/策略：长期保留，用户主动归档。
- 回测报告：默认保留 90 天，用户可"收藏"延长保留期。
- 盲测会话：长期保留（用户成长档案核心数据）。
- AI 对话：默认保留 30 天，敏感内容审核日志保留 180 天。

详见 [DATA_STORAGE.md](DATA_STORAGE.md)。

## 8. 可演进性

| 触发条件 | 演进动作 |
| --- | --- |
| 单服务流量超过单体阈值 | 按服务边界裂解为独立进程（已预留目录） |
| 行情数据量 > 200GB 或影响业务库 | 拆出独立行情库 / TimescaleDB |
| Worker 队列持续堆积 | 先增 Worker，再拆队列，最后评估 RabbitMQ |
| 多区域部署 | API 层就近接入 + 数据库异步副本 |
| 实时盲测（中后期）| 引入 SSE / WebSocket 通道与撮合服务 |

每一次演进都要新增 ADR，避免架构隐性漂移。

## 9. 反模式与禁区

- 不允许在 API 路由里直接写 SQL。
- 不允许在 Web 进程里跑 CPU 密集型回测/诊断同步任务。
- 不允许 AI 模块直接调用回测/诊断/策略写入接口。
- 不允许跨服务直接读对方表。
- 不允许把交易性决策能力开放给 AI（必须经 DSL）。
- 不允许在小程序端做需要保密的 K 线遮蔽逻辑（盲测可见窗口必须服务端裁剪）。
