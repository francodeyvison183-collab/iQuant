# 容量规划与性能目标

本文给出 iQuant MVP 阶段的容量假设、性能预算、资源规模与扩容路径。

当前目标：

> MVP 稳定支撑 1 万注册用户，并在业务增长时能平滑扩容到 10 万注册用户。

所有数字都是工程估算，必须通过压测和线上观测持续修正。

## 1. 用户规模假设

### 1.1 MVP 基线

| 指标 | MVP 目标 | 说明 |
| --- | --- | --- |
| 注册用户 | 10,000 | MVP 支撑目标 |
| 月活（MAU） | 3,000 ~ 5,000 | 30% ~ 50% 月活 |
| 日活（DAU） | 1,000 ~ 2,000 | 盘后和周末更集中 |
| 峰值活跃在线 | 200 ~ 500 | 微信小程序会话短、峰值集中 |
| 单用户日均会话 | 1 ~ 2 次 | 标注、盲测、报告查看 |
| 单会话动作数 | 20 ~ 50 次 | K 线切换、盲测推进、报告浏览 |

### 1.2 扩容规划上限

扩容路径按 10 万注册用户预留，但不在 MVP 阶段提前采购和部署全部资源。

| 指标 | 增长期目标 |
| --- | --- |
| 注册用户 | 100,000 |
| MAU | 30,000 |
| DAU | 8,000 ~ 12,000 |
| 峰值活跃在线 | 1,500 ~ 2,500 |

## 2. 流量预算

### 2.1 MVP API 峰值 QPS

| 路径 | MVP 峰值 QPS | 说明 |
| --- | --- | --- |
| 鉴权与会话维持 | 20 | 小程序登录、token 刷新 |
| K 线读取 | 40 | 命中 Redis / 本地缓存后回源较少 |
| 标注交互 | 20 | 标注点可批量提交 |
| 策略生成提交 | 2 | 重路径异步化 |
| 回测任务提交 | 2 | 仅创建任务和入队 |
| 回测结果查询 | 20 | 小程序轮询或拉取 |
| 盲测会话推进 | 60 | 高频操作，必须轻量 |
| 盲测操作提交 | 20 | 买入、卖出、观望 |
| 诊断报告查询 | 10 | 报告查看 |
| AI 对话 / 解释 | 5 | 成本高，配额限制 |
| 用户、设置、订阅 | 20 | 兜底 |
| **合计峰值** | **约 200 QPS** | 含缓冲 |

设计目标：

- 单 API 实例稳定支撑 100 QPS。
- MVP 常态 2 个 API 实例即可覆盖峰值。
- 高峰活动或增长期通过增加 API 实例横向扩容。

### 2.2 MVP 异步任务预算

| 任务 | MVP 峰值速率 | 单任务耗时目标 | 资源画像 |
| --- | --- | --- | --- |
| 策略生成 | 5 ~ 10 / 分钟 | 1 ~ 5 秒 | CPU 中 |
| 回测 | 10 ~ 20 / 分钟 | 5 ~ 30 秒 | CPU 高 |
| 执行诊断 | 20 ~ 40 / 分钟 | 1 ~ 10 秒 | CPU 中 |
| AI 解释 | 10 ~ 20 / 分钟 | 3 ~ 30 秒 | I/O 高、外部依赖 |
| 报告快照 | 10 ~ 20 / 分钟 | < 3 秒 | I/O 中 |
| 行情拉取 | 交易日盘后 | 数十秒 | I/O 高 |
| 数据清理 | 每日 | 数十秒 | I/O 中 |

MVP Worker 目标：

- 常态 2 个 Worker 容器。
- 高峰或活动期间扩到 4 个 Worker。
- CPU 重任务和 I/O 任务通过队列隔离，避免互相阻塞。

## 3. 性能目标

### 3.1 API 响应时间

| 路径 | P95 目标 | P99 目标 | 说明 |
| --- | --- | --- | --- |
| 鉴权刷新 | 100 ms | 300 ms | Redis + DB 轻读 |
| K 线读取（缓存命中） | 120 ms | 300 ms | Redis / 压缩数据 |
| K 线读取（回源） | 400 ms | 1 s | DB 查询 + 序列化 |
| 标注写入 | 200 ms | 600 ms | 简单写库 |
| 策略生成提交 | 200 ms | 500 ms | 仅校验 + 入队 |
| 回测提交 | 200 ms | 500 ms | 仅入队 |
| 回测查询 | 150 ms | 400 ms | 缓存优先 |
| 盲测推进 | 200 ms | 500 ms | 服务端裁剪 + 游标更新 |
| AI 对话提交 | 300 ms | 800 ms | 返回任务或流式会话句柄 |

原则：

- API 不同步执行回测、诊断、大模型调用。
- 任何 API P99 超过 2 秒必须进入性能看板和排查。

### 3.2 异步任务时延

| 任务 | MVP P95 目标 | 说明 |
| --- | --- | --- |
| 策略生成 | 15 秒 | 含排队 |
| 回测 | 60 秒 | 含排队；长区间回测需要限制 |
| 执行诊断 | 30 秒 | 含排队 |
| AI 解释 | 45 秒 | 含模型耗时，可降级 |
| 报告快照 | 10 秒 | - |

### 3.3 可用性目标

| 维度 | MVP 目标 |
| --- | --- |
| 整体 API 可用性 | 99.5% |
| 鉴权 / K 线读取 | 99.9% |
| 异步任务成功率 | 99% |
| 数据持久 | 业务数据不丢失，Redis 数据可重建 |

## 4. MVP 数据规模预算

| 数据 | MVP 估算量 | 备注 |
| --- | --- | --- |
| 用户表 | 10K 行 | 小 |
| 标的元数据 | 5K ~ 10K 行 | 小 |
| 日线 K 线 | 5K symbol × 20 年 ≈ 36M 行 | PostgreSQL 可承载 |
| 分钟 K 线 | MVP 只保留核心标的和有限周期 | 不做全市场高频 |
| 标注会话 + 标注点 | 5 会话/用户、20 点/会话 ≈ 1M 点 | 中 |
| 策略版本 | 5 个/用户 ≈ 50K 行 | 小 |
| 回测报告 | 10 次/用户 ≈ 100K 行元数据 | 明细放对象存储 |
| 盲测会话 | 10 次/用户 ≈ 100K 行 | 中 |
| 盲测操作 | 30 操作/会话 ≈ 3M 行 | 需要索引和归档策略 |
| 诊断报告 | 与盲测 1:1 ≈ 100K 行 | 中 |
| AI 消息 | 20 条/用户 ≈ 200K 行 | 可 30 天清理 |

结论：

- MVP 业务数据使用单 PostgreSQL 实例足够。
- 行情数据可先与业务库同实例、不同 schema 或表空间；当行情数据或查询压力明显影响业务库时再拆分。
- 对象存储用于回测明细、诊断明细和大报告，避免大 JSON 压垮主库。

## 5. MVP 资源规模估算

### 5.1 计算资源

| 角色 | MVP 建议规格 | 数量 | 说明 |
| --- | --- | --- | --- |
| API 容器 | 1 ~ 2 vCPU / 1 ~ 2 GB | 2 | 无状态，横向扩容 |
| Worker 容器 | 2 vCPU / 4 GB | 2 | 回测、诊断、AI 调用 |
| Beat 容器 | 0.5 vCPU / 512 MB | 1 | 定时任务，单实例 |
| 反向代理 / SLB | 云厂商托管 | 1 | TLS、WAF、限流 |

### 5.2 数据资源

| 服务 | MVP 建议规格 | 说明 |
| --- | --- | --- |
| PostgreSQL | 2 ~ 4 vCPU / 8 ~ 16 GB / SSD 200 GB | 业务 + MVP 行情 |
| Redis | 1 ~ 2 GB | 缓存、限流、Celery broker |
| 对象存储 | 按量 | 报告、快照、明细 |

### 5.3 成本原则

- MVP 不默认启用 Kubernetes。
- MVP 不默认启用 PostgreSQL 只读副本。
- MVP 不默认拆独立 TimescaleDB。
- MVP 不默认引入 RabbitMQ / Kafka。
- 所有扩容动作必须由监控数据触发。

## 6. 扩容路径

| 阶段 | 规模 | 主要形态 | 扩容动作 |
| --- | --- | --- | --- |
| Stage 0 内测 | 注册 < 1K | 单 API + 单 Worker + 托管 PG + Redis | 优先验证业务正确性 |
| Stage 1 MVP | 注册 1K ~ 10K | API ×2 + Worker ×2~4 + 单 PG + Redis | 支撑正式 MVP |
| Stage 2 增长 | 注册 10K ~ 30K | API/Worker 横向扩容 + Redis 主从 | 拆分 Worker 队列、优化慢查询 |
| Stage 3 放量 | 注册 30K ~ 100K | 引入只读副本、独立行情库、可选 K8s | TimescaleDB 拆出、HPA、任务队列治理 |
| Stage 4 成熟 | 注册 > 100K | 按瓶颈服务化 | RabbitMQ、分库、地域扩展、CDN 优化 |

## 7. 扩容触发条件

| 触发条件 | 动作 |
| --- | --- |
| API CPU > 60% 且 P95 > 300ms 持续 3 天 | 增加 API 实例 |
| Worker pending > 500 持续 10 分钟 | 增加 Worker 或拆队列 |
| 回测 P95 > 60s | 限制回测窗口、优化引擎、增加 CPU Worker |
| PostgreSQL CPU > 60% 或慢查询明显增加 | 补索引、改查询；仍不足再加只读副本 |
| 行情表 > 200GB 或影响业务查询 | 拆独立行情库 / TimescaleDB |
| Redis 内存 > 70% | 扩容 Redis 或分离 broker/cache |
| AI 成本 > 月预算 80% | 降级、缓存解释、限制配额 |

## 8. 性能治理约束

- 回测、诊断、AI 调用必须异步。
- 盲测“隐藏未来 K 线”的逻辑必须在服务端裁剪，不能依赖小程序端隐藏。
- 所有写操作必须幂等；重复提交不能产生重复标注、重复回测或重复诊断。
- 查询列表必须 keyset 分页，禁止深分页 `OFFSET`。
- 重任务必须有用户级并发限制。
- AI 必须有超时、配额和降级模板。
- 缓存必须有 TTL，Redis 不能作为唯一真理源。

## 9. 压测计划

MVP 上线前必须完成：

- API 核心路径压测：目标 200 QPS，P95 达标。
- 盲测推进压测：连续 step/action 场景，验证服务端裁剪和写入。
- 回测任务压测：20 / 分钟提交，验证 Worker 队列和 P95 完成时间。
- 诊断任务压测：40 / 分钟提交，验证诊断任务稳定性。
- AI 降级演练：模拟模型超时、失败和限流。
- 数据库慢查询检查：核心列表和详情接口必须有执行计划。

## 10. 已知风险

| 风险 | 描述 | 应对 |
| --- | --- | --- |
| 盲测推进高频写 | 用户连续点击导致写入放大 | 前端节流 + 服务端限流 + 批量事件 |
| 回测 CPU 长尾 | 长区间、多参数回测耗时过长 | 限制窗口、任务配额、分块优化 |
| 行情数据增长 | 行情表持续变大影响业务库 | 分区、归档、拆独立行情库 |
| AI 成本失控 | 用户高频对话和解释请求 | 日配额、缓存解释、模型分层 |
| 小程序端图表性能 | K 线过多导致卡顿 | 服务端分页/裁剪，端上只加载可视窗口 |
