# ADR-0008 模块化单体起步，按需服务化

## 背景

iQuant 后端有多个清晰的业务模块：策略、回测、盲测、诊断、AI、行情、用户等。常见的两种部署路径：

- 一开始就拆成多个微服务，独立部署、独立数据库。
- 模块化单体（modular monolith）：单进程部署，但代码强边界。

MVP 先支撑 1 万注册用户，同时预留增长到 10 万用户的路径。团队规模、业务正确性和运维复杂度是关键判断。

## 决策

MVP 与早期增长阶段采用模块化单体：

- 所有服务能力以 Python 包（`services/<name>-service`）形式存在，被 `apps/api` 与 `apps/worker` 引用。
- 数据库表按服务归属切分，跨服务不直读对方表，必须通过对方服务的用例函数。
- 依赖方向严格：`apps/* → services/* → packages/* → packages/domain`，CI 用 import-linter 强制。
- 任务、缓存、限流、监控按服务维度打标签，便于未来按线拆分。
- 当某一服务的资源占用、变更频率或团队规模触发明确阈值时，再裂解为独立部署单元。

## 影响

- 短期：减少多服务带来的运维负担与跨进程一致性问题，团队聚焦业务正确性。
- 中期：保留服务化逃生口；只有容量、团队或发布节奏触发明确阈值后才拆分。
- 强制约束：禁止任何跨服务读对方表的"便利写法"，否则未来拆解成本极高。

## 服务化触发条件（参考）

| 触发条件 | 动作 |
| --- | --- |
| 某服务 CPU/内存占用稳定 > 单体的 50% 且发布节奏与其他服务明显错配 | 拆为独立 Deployment + 独立镜像 |
| 某服务日变更频次显著高于平均，影响发布节奏 | 拆为独立部署单元 |
| 某服务依赖独特技术栈（GPU、特殊运行时） | 拆为独立部署单元 |
| 团队人数 > 10 且自然形成 2~3 个聚焦小组 | 按小组划分服务边界 |
| 某服务数据量增长触及单库瓶颈 | 拆库（先于拆服务） |

## 备选方案

- **一开始就拆微服务**：初期人员少时维护成本极高，跨服务事务和调试难度大，回报远低于代价。
- **不分模块的单体**：短期"快"，长期重写概率高，且本项目业务边界本就清晰，无理由不切。
- **进程内 + 进程间混合**：边界混乱，反而增加复杂度。
